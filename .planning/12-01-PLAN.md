## Overview

Phase 7 performed code-inspection-only UAT (Items 1-4). Phase 12 upgrades to **real browser testing** with keyboard navigation (Item 5) and full NVDA validation. This is a human-driven process: tester manually clicks through each checkpoint, verifies interactive demos work, tests keyboard navigation, and validates screen reader announcements.

**Why This Matters:**
- Phase 7 UAT relied on grep patterns and code review; missed runtime issues
- Phase 12 uses real browser + real screen reader (NVDA) to catch actual user experience problems
- Item 5 (keyboard navigation) requires hands-on testing; cannot be automated
- Screen reader announcements must be heard, not just syntactically correct

## Testing Procedure

### Pre-Test

1. **Setup NVDA:**
   - Start NVDA (insert + N or app launcher)
   - Enable focus mode (insert + spacebar)
   - Navigate to Chrome window

2. **Open checkpoint in Chrome:**
   - Click checkpoint link from index.html (sidebar navigation)
   - Wait 1 second for page to fully load
   - NVDA should announce: "Page heading: [Criterion Name]"

### Per-Checkpoint Testing (5 Items + 4 SR Checks)

#### **Item 1: Demo Toggle Works**
**Test:** Click Fail button, verify demo switches to fail mode; click Pass button, verify demo switches to pass mode.

1. Locate the demo container (first `.demo-container` on page)
2. Click the "FAIL" button (should have light gray background initially)
3. Visual verify: Demo shows fail example (e.g., bad form, missing labels, low contrast)
4. Click the "PASS" button
5. Visual verify: Demo shows pass example (e.g., correct form, good labels, sufficient contrast)
6. **Result:** P[ ] F[ ] | **Notes:** \_\_\_\_\_\_\_\_\_\_\_

If toggle button doesn't exist, is disabled, or doesn't change demo: **FAIL**

---

#### **Item 2: Code Blocks Sync**
**Test:** Code blocks show implementation for current demo mode (fail or pass).

1. After toggling to FAIL mode, scroll down to "Code Example" section
2. Visual verify: Code shows fail implementation (e.g., `<input>` without `label`, `<div role="button">`, etc.)
3. Toggle to PASS mode
4. Visual verify: Code shows pass implementation (e.g., `<input id="x">` + `<label for="x">`, native `<button>`, etc.)
5. Code should be visually distinct (different highlighted sections or entirely different code blocks)
6. **Result:** P[ ] F[ ] | **Notes:** \_\_\_\_\_\_\_\_\_\_\_

If code doesn't sync with demo mode or shows same code for both modes: **FAIL**

---

#### **Item 3: Announcements Functional**
**Test:** Screen reader announces demo state changes (fail/pass mode).

1. Start NVDA
2. Click Fail button in demo container
3. **NVDA should say:** "FAIL mode. [Announcement text from data-announce-fail attribute]" (e.g., "FAIL mode. Form label is missing")
4. Click Pass button
5. **NVDA should say:** "PASS mode. [Announcement text from data-announce-pass attribute]" (e.g., "PASS mode. Form has visible label")
6. Announcements must use **functional language** (user impact, not visual properties):
   - ✅ Good: "Form label is hidden from screen readers"
   - ❌ Bad: "Form label has a color of #f5f5f5 and a border of 2px solid #000"
7. **Result:** P[ ] F[ ] | **Notes:** \_\_\_\_\_\_\_\_\_\_\_

If announcements don't play, are silent, or use visual language (colors, hex codes, CSS properties): **FAIL**

---

#### **Item 4: WCAG Technique Codes Present**
**Test:** "How to Pass" section includes WCAG technique codes (G80, H32, etc.) with links to W3C.

1. Scroll to "How to Pass" or "Sufficient Techniques" section
2. Verify at least 2-3 technique codes visible (e.g., `G80: Providing a submit button to initiate a change of context`)
3. Click technique code link (blue underlined text)
4. **Should open:** W3C Understanding page (https://www.w3.org/WAI/WCAG22/Techniques/[code])
5. Technique page loads without error (200 OK, not 404)
6. **Result:** P[ ] F[ ] | **Notes:** \_\_\_\_\_\_\_\_\_\_\_

If no technique codes visible, links are broken (404), or links don't open W3C pages: **FAIL**

---

#### **Item 5: Keyboard Navigation Works**
**Test:** All interactive controls reachable and operable via keyboard (Tab, Enter, Space, Arrow keys).

**Keyboard Test Sequence:**

1. **Tab Order:** Press Tab repeatedly from top of page
   - Focus moves through all interactive elements (buttons, links, inputs, form controls)
   - Focus order is logical and matches visual left-to-right, top-to-bottom reading order
   - No elements skipped

2. **Demo Controls:**
   - Tab to "FAIL" button → Press Enter → Demo switches to fail mode
   - Tab to "PASS" button → Press Enter → Demo switches to pass mode
   - Verify button has visible focus outline (blue or high-contrast border)

3. **Form Controls (if present):**
   - Tab to `<input>`, `<select>`, `<textarea>` → Type/select value
   - Tab to `<button type="submit">` → Press Enter → Form submits (or relevant action triggers)
   - Verify all inputs have visible focus outlines

4. **Links:**
   - Tab to any `<a>` element → Press Enter → Link opens target page or section
   - Focus outline must be visible (not hidden by CSS)

5. **Focus Trap Check:**
   - Tab through entire page from top to bottom
   - Verify Tab key moves focus forward and does not loop back to top prematurely
   - Verify Shift+Tab moves focus backward smoothly
   - No "keyboard traps" (user gets stuck unable to Tab out)

6. **Result:** P[ ] F[ ] | **Notes:** \_\_\_\_\_\_\_\_\_\_\_

If any control cannot be reached via Tab, focus outline is invisible, or Tab order is illogical: **FAIL**

---

### Screen Reader Validation (NVDA, 4 Checks)

After completing Items 1-5, run 4 NVDA checks per checkpoint:

#### **SR Announce: State Changes**
1. Enable NVDA focus mode (Insert + Spacebar)
2. Navigate to demo toggle buttons (Tab key)
3. Activate Fail button (Enter key)
4. **Listen:** NVDA should announce "FAIL mode. [Functional announcement]"
5. Activate Pass button (Enter key)
6. **Listen:** NVDA should announce "PASS mode. [Functional announcement]"
7. **Result:** P[ ] F[ ] | **Notes:** \_\_\_\_\_\_\_\_\_\_\_

#### **SR Headings: Page Structure**
1. Press H to move to first heading on page
2. **Listen:** NVDA announces page heading (e.g., "Heading 1: 1.1.1 Non-text Content")
3. Press H repeatedly to move through all headings
4. **Verify:**
   - Headings are in logical order (H1 > H2 > H3, not H1 > H3)
   - All major sections have headings (Criterion Definition, Why This Matters, How to Test, etc.)
   - No skipped heading levels
5. **Result:** P[ ] F[ ] | **Notes:** \_\_\_\_\_\_\_\_\_\_\_

#### **SR Controls: Form Labels & Accessibility**
1. Press F to jump to first form control on page
2. **Listen:** NVDA announces label + control type (e.g., "Fail button, toggle")
3. Press F repeatedly through all form controls
4. **Verify:**
   - Every `<button>`, `<input>`, `<select>`, `<textarea>` has an associated label or aria-label
   - NVDA announces the label before/with the control
   - Control type is announced (button, textbox, radio, etc.)
   - No controls announced as "unlabeled" or "unavailable"
5. **Result:** P[ ] F[ ] | **Notes:** \_\_\_\_\_\_\_\_\_\_\_

#### **SR Live Region: Dynamic Updates**
1. Locate any aria-live region on page (visible or hidden)
2. If demo uses dynamic content updates (e.g., validation messages, status announcements):
   - Trigger the dynamic update (e.g., submit invalid form, change dropdown)
   - **Listen:** NVDA should announce the update without requiring Tab/navigation to reach it
   - Update should be announced immediately, not delayed
3. **Result:** P[ ] F[ ] | **Notes:** \_\_\_\_\_\_\_\_\_\_\_

If no aria-live regions exist on page, mark as N/A (not applicable).

## Failure Reporting & Fix Cycle

**When you find a failure:**

1. **Document immediately** in UAT-RESULTS.md (checkpoint ID, item number, specific failure description)
2. **Report to Claude** with:
   - Checkpoint ID (e.g., "1.1.1")
   - Failed item number (1-5 or SR check name)
   - Exact failure (e.g., "Button toggle button has no focus outline when tabbed")
   - Steps to reproduce
3. **Claude will:**
   - Read failure report
   - Fix the code (checkpoint HTML file)
   - Create atomic commit (one fix per commit)
   - Push to origin/dev
4. **You will:**
   - Pull latest dev: `git pull origin dev`
   - Re-test fixed checkpoint
   - Update UAT-RESULTS.md with P[ ] (pass) status
   - Continue testing remaining checkpoints

**Example Fix Cycle:**

```
TESTER: "Checkpoint 2.1.1 Keyboard — Item 5 fails. Button has focus ring
but it's invisible in light mode (blue on blue). Can't see focus."

CLAUDE: [Fixes focus outline color in assets/css/demo.css + button styles]
        [Commits: "fix(2-1-1): improve focus outline visibility"]
        [Pushes to origin/dev]

TESTER: [Pulls dev, re-tests checkpoint 2.1.1]
        "Fixed. Focus outline now visible. Item 5 passes."
        [Updates UAT-RESULTS.md: Item 5 - Keyboard Nav P[x]]
```
