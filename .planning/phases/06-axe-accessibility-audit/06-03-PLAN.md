## Common Fix Patterns (from RESEARCH.md)

| Issue Type | Fix Pattern | Example |
|-----------|-----------|---------|
| Missing alt text | Add alt attribute to img | `<img src="..." alt="description">` |
| Color contrast | Use CSS variables from base.css or adjust colors | `color: var(--color-text-default);` |
| Missing label | Add `<label for="id">` or aria-label | `<input id="name"> <label for="name">` |
| Missing heading | Add `<h2>`, `<h3>` with proper nesting | Heading structure A→B→B→C is invalid; use A→B→B→B |
| Focus visible | Apply outline-visible rule | `:focus-visible { outline: 3px solid; }` |
| ARIA violations | Fix aria-hidden, aria-label, role attributes | `role="region"`, `aria-live="polite"` |
| Semantic HTML | Replace div/span with semantic elements | Use `<button>` instead of `<div role="button">` |

## Accessibility Standards (WCAG 2.2 A/AA)
- 1.4.3 Contrast (Minimum) — 4.5:1 for normal text, 3:1 for large text
- 1.4.11 Non-text Contrast — 3:1 for UI components and graphical elements
- 2.4.7 Focus Visible — All keyboard-accessible elements must have visible focus indicator
- 4.1.2 Name, Role, Value — Form controls must have accessible name, role, and state
- 4.1.3 Status Messages — Dynamic updates must be announced to screen readers (aria-live)

Based on AUDIT-TRIAGE.md Real Issues table, fix each violation that is NOT deferred. Group fixes by type and checkpoint file for efficient commits.

Common fixes (reference RESEARCH.md for patterns):

**Color Contrast (1.4.3):**
- If text is too light on background, increase contrast by darkening text or lightening background
- Use CSS variables from base.css: `--color-text-default`, `--color-bg-default`
- For demo-related text, verify [data-theme="dark"] .demo-stage rule applies correctly
- Example fix: `color: var(--color-text-on-bg);` instead of hardcoded `#666`

**Focus Visible (2.4.7):**
- Add `:focus-visible { outline: 3px solid var(--color-focus); outline-offset: 2px; }` to any interactive element without visible focus
- Verify outline is visible on light AND dark mode
- Example: `button:focus-visible { outline: 3px solid var(--color-focus); }`

**Alt Text (1.1.1):**
- Add `alt="description"` to all `<img>` tags
- Alt text should describe the image content or function, not "image of..."
- Example: `<img src="logo.svg" alt="WCAG Demo Site Logo">`

**Semantic HTML (4.1.2):**
- Replace `<div class="button">` with `<button>`
- Replace `<div role="button">` with `<button>`
- Ensure all form controls have associated `<label>` elements
- Example: `<label for="name">Name:</label><input id="name" type="text">`

**ARIA Labels (4.1.2):**
- Add `aria-label` for icon-only buttons
- Add `aria-describedby` for complex form controls
- Add `aria-live="polite"` for dynamic content updates
- Verify `aria-hidden="true"` is only on decorative elements

**Status Messages (4.1.3):**
- For demo announcements, ensure `aria-live="polite"` or `aria-live="assertive"` is present
- Verify demo-toggle.js correctly fires announcements for state changes

Workflow:
1. For each violation in AUDIT-TRIAGE.md Real Issues:
   - Locate the file and element (from "Location" column)
   - Identify the fix pattern (contrast, focus, label, ARIA, etc.)
   - Apply the fix to the HTML/CSS
   - Verify fix doesn't break demo functionality
   - Test in browser: light mode and dark mode

2. Commit strategy:
   - Group fixes by type: one commit per fix type (e.g., "fix(a11y): improve color contrast in 5 checkpoints")
   - Include in commit message: number of files fixed, which WCAG criteria addressed
   - Example: `fix(a11y): add focus outline to interactive elements (2.4.7 Focus Visible) — 8 files`

3. Do NOT fix:
   - Violations in [data-mode="fail"] containers (intentional teaching examples)
   - Violations in [data-mode="pass"] containers marked "DEFERRED" in AUDIT-TRIAGE.md
   - Any violations marked "Intentional" in AUDIT-TRIAGE.md

Expected result: All real, non-deferred issues fixed. Each commit message documents which WCAG criteria and how many files.
