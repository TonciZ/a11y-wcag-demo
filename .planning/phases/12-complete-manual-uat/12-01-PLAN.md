    **Testing Workflow (Tester Instructions):**

    1. **Setup:**
       - Ensure Chrome browser and NVDA screen reader are both available
       - Open .planning/UAT-RESULTS.md in editor for updating test results
       - Keep CHECKLIST.md open in a second window for reference

    2. **Per-Checkpoint Testing (44 total, ~5-10 min per checkpoint = 200-250 min total):**
       - Open checkpoint in Chrome: visit http://localhost:8080/checkpoints/{checkpoint-name}.html (or file:// if no server)
       - **Items 1-5 (Visual + Keyboard):**
         1. Item 1: Click toggle button, verify aria-pressed updates, demo state changes, correct announcement spoken to aria-live region
         2. Item 2: Verify code block text changes synchronously with toggle state
         3. Item 3: Search announcement text (both fail and pass) for visual properties (colors, hex codes, contrast ratios, CSS properties). Should describe user impact only.
         4. Item 4: Verify technique codes visible on page (H-series, F-series, ARIA-series)
         5. Item 5: Tab from before the demo → through demo → after demo; verify no trap, logical left-to-right/top-to-bottom order, focus visible on all interactive elements
       - **NVDA Screen Reader Checks (4 checks):**
         1. SR Announce: Activate NVDA, toggle demo, verify data-announce-fail/pass text is spoken
         2. SR Headings: Use NVDA H-key navigation, verify heading structure is logical (no skipped levels, proper hierarchy)
         3. SR Controls: NVDA reads toggle button name/role correctly (e.g., "Show pass example, button, pressed" or "Show fail example, button, not pressed"), inputs/links announced with correct roles
         4. SR Live Region: Toggle demo with NVDA on, verify aria-live region announces state change without user pressing Tab or moving focus

    3. **Failure Reporting:**
       - For each failing item, mark `[!]` in UAT-RESULTS.md
       - Add brief description in Notes column (e.g., "Keyboard trap in input fields", "Announcement includes color name", "Focus outline not visible on button")
       - Report ONE checkpoint at a time via chat message, listing all failures found
       - Wait for Claude to discuss fix approach before testing next checkpoint
       - Flag fail-mode demos separately — they are intentionally broken for educational purposes, NOT bugs (example: 2.1.2 No Keyboard Trap shows a trap in fail code; that's correct behavior, not a fix needed)

    4. **Passing Checkpoint:**
       - Mark all items as `[x]` if no failures found
       - Set Notes to "All items pass, no issues found" or leave blank

    5. **After All 44 Tested:**
       - Update UAT-RESULTS.md summary table with total counts
       - Report all failures to Claude in batch (or incrementally as testing progresses)
       - Claude will implement fixes after reviewing all failures
       - Tester re-tests fixed checkpoints to verify closes

<task type="auto">
  <name>Task 3: Implement fixes for UAT failures and commit</name>
  <files>checkpoints/*.html, assets/js/*.js, assets/css/*.css</files>
  <action>
    After human tester completes testing of all 44 checkpoints and reports all failures via chat, Claude implements fixes based on failure descriptions.

    **Fix Scope:**
    - Real issues in pass-mode demos and site chrome (header, nav, buttons, etc.)
    - Real issues in NVDA announcements and screen reader behavior
    - Real issues in keyboard navigation and focus management
    - EXCLUDE: Fail-mode demos are intentionally broken for educational purposes; no fixes for intentional failures

    **Fix Strategy by Issue Type:**

    1. **Keyboard Navigation Issues (Tab order, focus trap, focus visibility):**
       - Review checkpoint HTML tab order; ensure follows visual left-to-right, top-to-bottom
       - Check for focus trap using dev tools (open checkpoint, Tab from start to end, verify can exit all interactive areas)
       - Add/verify focus-visible CSS if focus outline not showing
       - Update tabindex if needed (use sparingly; semantic HTML usually handles correct order)
       - Verify logical order: demo toggle → demo inputs/controls → code block toggle (if any) → next section

    2. **NVDA Announcement Issues:**
       - Search checkpoint for data-announce-fail/pass attributes containing: color names, hex codes (#FFFFFF), contrast ratios ("X to 1"), CSS property names (border, background, color, font-weight)
       - If found, rewrite announcement to describe user impact instead (refer to CHECKLIST.md Item 3 for example patterns)
       - Example fix: "Form controls have 4.5:1 contrast ratio" → "Form controls have clearly visible boundaries"
       - After fix, verify aria-live region still fires correctly (check demo-toggle.js)

    3. **Heading Structure / NVDA Heading Navigation Issues:**
       - Review checkpoint heading levels using NVDA (H-key navigation shows structure)
       - Ensure no skipped levels (h1 → h2, not h1 → h4)
       - Fix if checkpoint intro is h2 when page starts with h1; adjust hierarchy to match page structure

    4. **Demo Toggle / Code Block Sync Issues:**
       - Verify demo-toggle.js is loaded and functioning (inspect Elements in dev tools)
       - Check data-mode attribute is toggling correctly (use dev tools Inspector, toggle button, watch attribute change)
       - Verify code block is using data-mode selector in CSS to show/hide correct code
       - If code block not syncing: may need to add missing CSS rule in assets/css/demo.css

    5. **Technique Codes Not Visible:**
       - Search checkpoint for technique code text (H-series, F-series, ARIA-series)
       - If found but not visible, check for display:none or hidden class
       - Remove hiding; make technique codes visible by default

    6. **Site Chrome Issues (Header, Navigation, Buttons):**
       - Refer to assets/css/layout.css and assets/css/base.css for existing style patterns
       - Fix contrast issues using existing CSS variables (check assets/css/base.css for --color-* tokens)
       - Fix focus visible on buttons: add focus-visible outline rule
       - Fix keyboard trap in sidebar nav: verify navigation doesn't trap focus (usually handled by semantic HTML structure)

    **Commit Strategy (Batch, Grouped by Issue Type):**
    After all issues fixed, create atomic commits grouped by issue category:

    Example commits:
    ```
    fix(keyboard-nav): correct tab order in [checkpoint-list] to follow visual reading order
    fix(announcements): remove visual property language from NVDA announcements in [checkpoint-list]
    fix(chrome): add focus-visible outline to demo toggle buttons
    fix(headings): adjust h2/h3 hierarchy in [checkpoint] to match page structure
    ```

    Each commit message references:
    - Issue type (keyboard-nav, announcements, chrome, headings, code-sync, technique-codes)
    - Affected checkpoints or file
    - Brief description of fix
    - Include [UAT-01], [UAT-02], [UAT-03] requirement references if appropriate

    **Verification After Fixes:**
    - Human tester re-tests fixed checkpoints (run Items 1-5 + NVDA checks again on fixed checkpoints)
    - Mark re-tested items as `[x]` in UAT-RESULTS.md
    - If new issues found, iterate (report → fix → re-test)
    - Final state: all 44 checkpoints pass all items
  </action>
