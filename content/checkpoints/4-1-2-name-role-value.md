## Description

For all user interface components - including form elements, links, and custom widgets created with scripting - the following must be programmatically determinable or settable: the name and role of the component, and any states, properties, and values that can be set by the user. This criterion ensures that assistive technologies such as screen readers can identify what a component is (its role), what it is called (its name), and what state it is in (e.g., expanded, selected, checked). It is the foundation of accessible rich internet applications and applies to every interactive element on a page.

## Fail Explanation

A custom button built from a `<div>` or `<span>` element that has no `role` attribute and no accessible name fails this criterion, because a screen reader has no way to identify it as a button or know what it does. Similarly, a custom accordion widget that expands and collapses content but does not use `aria-expanded` to communicate its state leaves screen reader users unable to know whether the panel is open or closed. Using images as buttons without `alt` text, or linking icons without accessible names, are also very common failures.

## Pass Explanation

A component passes when it has a programmatically determinable accessible name (from visible label text, `aria-label`, `aria-labelledby`, or an `alt` attribute), a correct semantic role (either from native HTML semantics or an explicit ARIA `role`), and accurate state information conveyed via ARIA attributes (such as `aria-expanded`, `aria-checked`, `aria-selected`, `aria-pressed`). Native HTML elements like `<button>`, `<input>`, `<select>`, and `<a href>` satisfy these requirements when used correctly with proper labels; custom widgets require explicit ARIA and scripting to achieve the same result.

## How To Test

1. Using browser developer tools or an accessibility tree inspector (e.g., Firefox Accessibility panel), examine every interactive element on the page.
2. Confirm each element has a non-empty accessible name (visible label, `aria-label`, `aria-labelledby`, or `alt` text for images).
3. Confirm each element has an appropriate role - either from native HTML semantics (button, link, checkbox) or an explicit `role` attribute.
4. For stateful components (accordions, toggles, tabs, checkboxes, comboboxes), confirm the current state is communicated through the appropriate ARIA attribute.
5. Test with a screen reader (e.g., NVDA + Chrome or VoiceOver + Safari): navigate to each interactive control and confirm its name, role, and state are announced correctly.
6. Activate interactive controls (open accordions, toggle switches, select tabs) and confirm the screen reader announces the updated state.
7. Check that any custom widgets built with `<div>`, `<span>`, or `<a>` elements that function as buttons or other controls carry the necessary `role` and `aria-*` attributes.

## Notes

This criterion specifically covers author-created UI components and does not apply to platform-level user agent controls that the author cannot modify. The easiest way to satisfy this criterion is to use native, semantic HTML elements (which have built-in name, role, and value semantics) rather than building custom widgets from generic elements, which require additional ARIA and scripting to achieve the same accessibility outcome.

# WCAG 4.1.2 Name, Role, Value: complete research for interactive demos

**WCAG 2.2 Success Criterion 4.1.2 requires that every user interface component exposes three things to assistive technology: a programmatic name, a semantic role, and its current state or value.** This Level A criterion is the single most foundational rule governing how custom and standard controls communicate with screen readers, voice input software, and other AT. Failures here are among the most common accessibility defects found in the wild — the WebAIM Million analysis consistently ranks missing form labels, empty buttons, and empty links among the top five homepage errors. The research below covers everything needed to build at least three distinct, practical HTML/CSS/JS demos that illustrate different failure types with toggleable passing and failing states.

---

## What "Name," "Role," and "Value" actually mean

The normative text of SC 4.1.2 reads: *"For all user interface components (including but not limited to: form elements, links and components generated by scripts), the name and role can be programmatically determined; states, properties, and values that can be set by the user can be programmatically set; and notification of changes to these items is available to user agents, including assistive technologies."*

Each term has a precise W3C definition that differs from everyday developer usage:

**Name** is the text by which software identifies a component to the user — the "accessible name." This is unrelated to the HTML `name` attribute. The accessible name is computed via a strict priority cascade: `aria-labelledby` (highest) → `aria-label` → native labeling (`<label>`, `<legend>`, `<caption>`, `alt`, text content) → `title` attribute → `placeholder` (lowest, not recommended). A `<button>` gets its name from its text content; an `<input>` gets its name from an associated `<label>`; an icon button with no text and no `aria-label` has **no name at all**, which is a failure.

**Role** is the function of the component — button, link, checkbox, tab, slider, textbox, and so on. Native HTML elements carry implicit ARIA roles automatically: `<button>` = `button`, `<a href>` = `link`, `<input type="checkbox">` = `checkbox`, `<select>` = `combobox`. A bare `<div>` or `<span>` has the `generic` role, meaning assistive technology treats it as non-interactive static content regardless of click handlers attached to it. Custom controls built from `<div>` elements must receive an explicit `role` attribute.

**Value** (and states/properties) covers dynamic characteristics: `aria-checked` on checkboxes, `aria-expanded` on disclosure triggers, `aria-selected` on tabs, `aria-pressed` on toggle buttons, `aria-valuenow` on sliders. Native HTML form elements expose these automatically (a checked `<input type="checkbox">` reports "checked" to AT). Custom widgets must manage ARIA states in JavaScript and keep them synchronized with the visual presentation.

The SC applies at **Level A** — the most fundamental conformance tier. A note clarifies: *"This success criterion is primarily for web authors who develop or script their own user interface components. Standard HTML controls already meet this success criterion when used according to specification."*

## The eight most common real-world failure patterns

These patterns emerge repeatedly in audits, bug reports, and the WebAIM Million data. Each maps to one or more official W3C failure techniques.

**1. Icon buttons without accessible names (F68, axe rule `button-name`).** A `<button>` containing only a Font Awesome `<i>`, an SVG, or a CSS-generated icon with no visible text and no `aria-label`. Screen readers announce simply "button" with no indication of purpose. This is the single most frequently reported 4.1.2 failure. Hamburger menu buttons, close buttons, search icons, and social media icon links are classic offenders.

**2. Form inputs without associated labels (F68, axe rule `label`).** An `<input>` preceded by adjacent text like "Email:" that is not wrapped in a `<label>` and has no `for`/`id` association, `aria-label`, or `aria-labelledby`. NVDA announces "edit, blank." JAWS may heuristically guess a label from nearby text, which masks the failure during audits but does not constitute a programmatic association.

**3. Custom controls built from `<div>`/`<span>` without roles (F59, F15).** A `<div onclick="...">` acting as a button, checkbox, or dropdown. Without `role="button"` (or the appropriate role), the element is invisible to AT as an interactive control. The W3C failure technique F59 specifically calls out "using script to make div or span a user interface control in HTML without providing a role for the control."

**4. Missing ARIA states on custom widgets (F15, F79).** A custom toggle switch that changes visually but never sets `aria-checked`. A custom accordion header that expands a panel but never updates `aria-expanded`. An ARIA tab interface where `aria-selected` is never toggled. AT users perceive a frozen, static control even as sighted users see state changes.

**5. Image-only links with no alt text (F89, axe rule `link-name`).** An `<a>` wrapping an `<img>` with empty or missing `alt`. Screen readers fall back to announcing the filename ("user-avatar-dot-png, link") or just "link, graphic."

**6. Custom tabs without tablist/tab/tabpanel semantics.** Tabs built from styled `<div>` elements with click handlers but no `role="tablist"`, `role="tab"`, `role="tabpanel"`, or `aria-selected`. AT users encounter what appears to be a series of generic text elements with no indication of tab structure, selected state, or tab count.

**7. Multi-part form fields with only one label (F86).** A phone number split into three `<input>` fields where only the first has a label ("Phone number"). The second and third fields announce as "edit, blank" — users cannot determine what each field expects.

**8. Dynamic content changes without notification (F20).** Non-text content (images, icons) updated via script without updating the text alternative, or status messages injected into the DOM without `aria-live` regions.

## Sufficient techniques and how they work

The W3C organizes sufficient techniques for 4.1.2 into four situations:

**Situation A — Standard HTML controls.** The primary techniques are **H91** (using HTML form controls and links as intended), **H44** (using `<label>` with `for`/`id` to associate text with form controls), **H65** (using `title` when visible labels cannot be used), **H64** (using `title` on `<iframe>`), and **H88** (using HTML according to spec). The key insight: if you use `<button>`, `<a href>`, `<input>`, `<select>`, and `<textarea>` correctly with proper labels, you automatically pass 4.1.2.

**Situation B — Repurposing HTML elements with ARIA.** Techniques include **ARIA14** (using `aria-label` for invisible labels, e.g., a close button: `<button aria-label="Close">×</button>`), **ARIA16** (using `aria-labelledby` to point to one or more text elements on the page), and **G108** (using markup features to expose name and role).

**Situation D — Creating entirely custom components.** Techniques include **ARIA4** (using a WAI-ARIA role to expose the role), **ARIA5** (using state and property attributes to expose state), and **ARIA16** again for naming. The APG (Authoring Practices Guide) provides complete patterns for custom buttons, checkboxes, comboboxes, tabs, dialogs, accordions, sliders, trees, and menus — each specifying required roles, required states, keyboard interaction, and focus management.

## How screen readers expose these failures

Screen reader behavior varies in ways that matter for testing and demos:

**NVDA** is the strictest. An unlabeled button announces "button" — nothing more. An unlabeled input announces "edit, blank." A `<div>` with a click handler is treated as static text in browse mode and is completely skipped in focus mode. NVDA does not apply heuristics to guess labels, making it the most reliable tool for auditing.

**JAWS** applies heuristic label inference. It may announce adjacent text as an input's label even without a programmatic `<label>` association. This masks F68 failures during testing — a critical gotcha for auditors. JAWS also auto-switches between virtual cursor and forms mode based on element roles; a `<div>` without a role won't trigger forms mode.

**VoiceOver (macOS/iOS)** announces unlabeled buttons as "button" or "button, untitled." Unlabeled inputs get "text field" with no context. Custom controls without roles are announced as plain text. VoiceOver on iOS adds "double-tap to activate" for focusable elements even without role context, which can confuse users.

**TalkBack (Android)** announces unlabeled buttons as "button, unlabeled" and may add "double-tap to activate." Unfocusable `<div>` controls may not be reachable at all in linear navigation.

The key cross-reader finding: **state changes on custom controls without ARIA state attributes are universally silent.** If a toggle switch flips visually but `aria-checked` is not updated, all four major screen readers announce nothing on activation beyond "button" (if it even has a button role). This makes missing ARIA states an excellent demo scenario — it's dramatic and immediately understandable.

## What testers should check and how

Manual testing for 4.1.2 follows a systematic pattern:

**Step 1: Inspect the accessibility tree.** In Chrome DevTools (Elements → Accessibility pane) or Firefox (Accessibility Inspector), verify every interactive element has a non-empty Name, an appropriate Role, and current State values. This is the fastest way to detect failures without a screen reader.

**Step 2: Tab through all interactive elements.** Every control must be keyboard-reachable. Custom controls must respond to expected keys (Enter/Space for buttons, Space for checkboxes, arrow keys within tab lists). If tabbing skips a visually interactive element, it likely lacks `tabindex` and may lack a role.

**Step 3: Screen reader testing.** Use NVDA + Chrome/Firefox on Windows and VoiceOver + Safari on macOS. For each interactive element, verify: (a) the name is announced and describes the purpose, (b) the role is announced (button, checkbox, tab, link, etc.), (c) the state is announced (checked, expanded, selected), and (d) state changes are announced upon interaction.

**Step 4: Automated scans.** Run axe DevTools, WAVE, or Lighthouse. Key axe rules for 4.1.2 include `button-name`, `link-name`, `label`, `select-name`, `input-image-alt`, `aria-command-name`, `aria-input-field-name`, `aria-toggle-field-name`, `aria-roles`, `aria-required-attr`, and `aria-valid-attr-value`. **Automated tools catch roughly 57% of 4.1.2 issues** — they reliably find missing names and invalid ARIA but cannot verify that names are meaningful, that states update dynamically, or that keyboard patterns follow APG expectations.

## Developer fix patterns: the code transformations

The fundamental developer guidance for 4.1.2 centers on one principle: **use native HTML first, ARIA only when native semantics are insufficient.** This is Rule 1 of the five Rules of ARIA Use. Native `<button>`, `<input>`, `<select>`, and `<a href>` elements provide built-in role, keyboard behavior, focus management, and state exposure. ARIA adds only semantics — never behavior.

When custom components are unavoidable, developers must provide all three layers manually: (1) `role` attribute matching the widget pattern, (2) accessible name via `aria-label` or `aria-labelledby`, and (3) state attributes updated in JavaScript on every interaction. The APG pattern pages are the authoritative reference for which roles, states, and keyboard interactions each widget type requires.

A critical subtlety: **`aria-label` only works on elements whose role supports naming.** Using `aria-label` on a bare `<div>` or `<span>` without an explicit role has no effect — the label is silently ignored. And `aria-label` on elements that derive name from content (like `<button>`) **replaces** the visible text for AT users, which can create a mismatch violating SC 2.5.3 (Label in Name) if the visible text isn't contained within the `aria-label` value.

## Five proposed demo scenarios for the static site

Based on the research, the following demos cover distinct failure types, are visually compelling, and provide clear before/after contrasts. Each demo should present a "Failing" state (default) and a "Passing" state toggled by the user.

### Demo 1: Icon buttons — the "Name" problem

This demo isolates the **accessible name** requirement. Present a toolbar of common icon buttons (search, menu/hamburger, close, favorite/heart, share) in two versions:

**Failing state:**
```html
<button class="icon-btn"><i class="fa fa-search"></i></button>
<button class="icon-btn"><svg aria-hidden="true"><!-- hamburger icon --></svg></button>
<button class="icon-btn">×</button>  <!-- close button with just × character -->
```
Screen reader output panel shows: "button" … "button" … "button, times" — no purpose communicated.

**Passing state:**
```html
<button class="icon-btn" aria-label="Search">
  <i class="fa fa-search" aria-hidden="true"></i>
</button>
<button class="icon-btn" aria-label="Main menu" aria-expanded="false">
  <svg aria-hidden="true"><!-- hamburger --></svg>
</button>
<button class="icon-btn" aria-label="Close dialog">×</button>
```

**Teaching points:** What `aria-label` does, why `aria-hidden="true"` on decorative icons matters, accessible name computation priority, the axe `button-name` rule.

**Techniques demonstrated:** ARIA14, F68 (failure), F89 (for image-only variant).

### Demo 2: Custom toggle switches — the "Role" and "Value" problem

This demo isolates **role and state** requirements. Present a settings panel with three toggle switches for "Dark mode," "Notifications," and "Auto-save":

**Failing state:**
```html
<div class="toggle-row">
  <span>Dark mode</span>
  <div class="toggle" onclick="toggle(this)">
    <div class="toggle-thumb"></div>
  </div>
</div>
```
The `<div>` has no role, no `tabindex`, no `aria-checked`, and no accessible name. Screen readers skip it entirely or announce only "Dark mode" as static text.

**Passing state:**
```html
<div class="toggle-row">
  <span id="dm-label">Dark mode</span>
  <button role="switch" aria-checked="false" aria-labelledby="dm-label"
          onclick="this.setAttribute('aria-checked',
            this.getAttribute('aria-checked')==='true'?'false':'true')">
    <span class="toggle-thumb"></span>
  </button>
</div>
```

**Teaching points:** Why `<div>` is invisible to AT as an interactive control (F59), what `role="switch"` communicates, how `aria-checked` exposes state, keyboard operability via `<button>`, the difference between NVDA's strict announcement and JAWS's heuristics.

**Techniques demonstrated:** ARIA4, ARIA5, F59 (failure), F15 (failure).

### Demo 3: Custom tab interface — the compound widget problem

This demo shows a **complete widget pattern** where role, name, value, AND interaction all matter. Present a three-tab interface (e.g., "Overview," "Features," "Pricing"):

**Failing state:**
```html
<div class="tabs">
  <div class="tab active" onclick="showTab(0)">Overview</div>
  <div class="tab" onclick="showTab(1)">Features</div>
  <div class="tab" onclick="showTab(2)">Pricing</div>
</div>
<div class="panel">Overview content...</div>
```
No `role="tablist"`, no `role="tab"`, no `role="tabpanel"`, no `aria-selected`, no `aria-controls`/`aria-labelledby` associations, no arrow-key navigation. AT users encounter three generic text elements with no semantic structure.

**Passing state:**
```html
<div role="tablist" aria-label="Product information">
  <button role="tab" id="tab-1" aria-selected="true" aria-controls="panel-1">Overview</button>
  <button role="tab" id="tab-2" aria-selected="false" aria-controls="panel-2" tabindex="-1">Features</button>
  <button role="tab" id="tab-3" aria-selected="false" aria-controls="panel-3" tabindex="-1">Pricing</button>
</div>
<div role="tabpanel" id="panel-1" aria-labelledby="tab-1">Overview content...</div>
<div role="tabpanel" id="panel-2" aria-labelledby="tab-2" hidden>Features content...</div>
<div role="tabpanel" id="panel-3" aria-labelledby="tab-3" hidden>Pricing content...</div>
```
With JavaScript managing `aria-selected`, `tabindex` roving, and arrow-key navigation per the APG tabs pattern.

**Teaching points:** How compound widgets require multiple coordinated ARIA attributes, the APG pattern as the authoritative reference, roving `tabindex` for keyboard navigation within a widget, `aria-controls`/`aria-labelledby` associations, how screen readers announce "Tab 1 of 3, selected."

**Techniques demonstrated:** ARIA4, ARIA5, ARIA16, G108, F59 (failure).

### Demo 4 (bonus): Form inputs — the label association problem

Present a registration form with three fields (name, email, password):

**Failing state:** Plain text adjacent to inputs with no `<label>`, no `for`/`id`, no ARIA. Optionally include a variant with `<label>` present but `for` attribute missing.

**Passing state:** Proper `<label for="id">` associations. Include a variant showing `aria-labelledby` for a multi-part field (e.g., date with month/day/year inputs in a `role="group"` with a group label).

**Teaching points:** F68, H44, the difference between visual proximity and programmatic association, why JAWS may mask the failure via heuristic guessing, how axe `label` rule works.

### Demo 5 (bonus): Image-only links — the missing name on links

Present a row of social media icon links and a logo link:

**Failing state:** `<a href="..."><img src="twitter.svg" alt=""></a>` — empty `alt` on the only content in the link.

**Passing state:** Meaningful `alt` text or `aria-label` on the link.

**Teaching points:** F89, how screen readers announce filenames as fallback, the `link-name` axe rule.

## The accessible name computation priority — a reference table for the demos

| Priority | Source | Example | Notes |
|---|---|---|---|
| 1 (highest) | `aria-labelledby` | `<input aria-labelledby="lbl">` | Can reference multiple IDs; works on hidden elements; overrides everything |
| 2 | `aria-label` | `<button aria-label="Close">×</button>` | Invisible string; replaces child text for AT; only works on labelable elements |
| 3 | Native labeling | `<label for="x">Email</label><input id="x">` | `<label>`, `<legend>`, `<caption>`, `alt`, text content |
| 4 | `title` attribute | `<input title="Search">` | Fallback only; not visible to keyboard users; inconsistent support |
| 5 (lowest) | `placeholder` | `<input placeholder="Email">` | Disappears on input; poor contrast; never recommended as sole name |

## The five Rules of ARIA Use — essential context for all demos

Each demo should reference these rules in its educational content:

1. **Don't use ARIA if native HTML works.** Native `<button>`, `<input type="checkbox">`, `<select>` provide role, name, value, and keyboard behavior automatically.
2. **Don't change native semantics unless necessary.** Don't write `<h2 role="tab">` — wrap instead: `<div role="tab"><h2>...</h2></div>`.
3. **All interactive ARIA controls must be keyboard-usable.** `role="button"` on a `<div>` requires `tabindex="0"` and Enter/Space handlers. ARIA adds semantics, never behavior.
4. **Don't use `role="presentation"` or `aria-hidden="true"` on focusable elements.** This creates ghost elements that receive focus but announce nothing.
5. **All interactive elements must have an accessible name.** This is SC 4.1.2's "Name" requirement, distilled to its essence.

## Axe-core rules that map to SC 4.1.2

For the demo page's "automated testing" educational section, these are the key rules testers should know:

- **`button-name`** — every `<button>` and `role="button"` element needs a non-empty accessible name
- **`link-name`** — every `<a href>` and `role="link"` element needs a non-empty accessible name
- **`label`** — every `<input>`, `<textarea>`, `<select>` needs an associated label
- **`select-name`** — `<select>` elements specifically must have names
- **`aria-command-name`** — ARIA buttons, links, and menuitems need names
- **`aria-input-field-name`** — ARIA textboxes, comboboxes, listboxes, searchboxes, spinbuttons need names
- **`aria-toggle-field-name`** — ARIA checkboxes, radios, switches, menuitemcheckboxes need names
- **`aria-roles`** — role values must be valid WAI-ARIA roles
- **`aria-required-attr`** — required ARIA attributes must be present (e.g., `aria-checked` on `role="checkbox"`)
- **`aria-valid-attr-value`** — ARIA attribute values must be valid (e.g., `aria-checked="yes"` is invalid; must be `"true"`)

## Conclusion: from research to demo architecture

The three core demos should each isolate a different axis of the Name-Role-Value triad. **Demo 1 (icon buttons) teaches "Name"** — what happens when AT cannot identify a control's purpose. **Demo 2 (toggle switches) teaches "Role" and "Value"** — what happens when AT cannot determine what kind of control it is or what state it is in. **Demo 3 (tabs) teaches the compound case** — a complete widget pattern where name, role, value, keyboard interaction, and focus management must all work together. The two bonus demos (form labels and image-only links) round out coverage of the most common automated audit findings.

Each demo should include: a live failing example, a live passing example, a toggle to switch between them, a simulated screen reader output panel showing what each version announces, the relevant WCAG technique and failure codes, and the axe-core rule that would flag the failure. This structure gives testers the "what to look for" and developers the "how to fix it" — directly fulfilling the dual-audience requirement.
